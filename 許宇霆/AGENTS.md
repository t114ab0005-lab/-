# AGENTS.md

## Project Overview
本專案是「CampusShift (時間變現曆)」的前端 MVP 系統。
這是一個針對大學生零碎時間變現的媒合平台。
專案範圍嚴格限制：
- **純前端架構**：包含「任務大廳」、「一鍵空檔速配」、「發布任務 (Demo)」、「個人資料」四個主要畫面的 UI 介面與前端互動邏輯。不需要實作登入系統、真實金流與真實 GPS。
- **無後端資料庫**：「發布任務」功能僅作前端模擬（將新資料 push 進 JavaScript 陣列中），絕對不開發真實後端與資料庫。
- **UI 生成**：畫面全數由 Google Stitch 產出，工程師僅負責串接邏輯與分頁切換。

## Tech Stack
本專案的 UI 骨架與樣式主要由 Google Stitch 生成，請嚴格遵守以下技術限制：
- **HTML**: HTML5 (請保留 Stitch 生成的 DOM 結構與 Tailwind class，勿隨意大幅改動容器架構以免跑版)。
- **CSS**: Tailwind CSS via CDN (完全依賴 Tailwind Utility classes 進行排版，**不額外手寫自訂的 CSS 檔案**)。
- **JavaScript**: Vanilla JavaScript (原生 JS，ES6+ 語法。嚴禁引入 React, Vue 或 jQuery 等框架)。
- **分頁架構**: 使用 Vanilla JS 控制 DOM 的顯示/隱藏 (display: none/block) 來實作單頁應用 (SPA) 的四個分頁切換。
- **任務媒合實作方式**: 透過 Vanilla JS 的 Array `filter` 與 `includes` 方法實作標籤比對。
- **資料結構演算法**: 以原生 JS 實作 Priority Queue (優先權佇列)。

## File Structure (配合 Stitch 產出簡化)
請維持以下檔案結構，不要隨意新增後端或樣式檔案：
- index.html (由 Stitch 生成的主入口，包含導覽列與四個分頁的 HTML 結構。工程師需新增必要的 `id` 屬性以利 JS 綁定)
- script.js (包含：分頁切換邏輯、標籤篩選機制、Priority Queue 演算法、資料渲染)
- data/
  - jobs.js (存放預設的假任務 Mock Data 陣列)
- AGENTS.md (本文件)
- PRD.md (產品需求文件)
- USER_STORY.md (使用者故事)

## Must Have (核心功能與開發規範)
1. **導覽與分頁**：實作流暢的 Navbar/Tab 切換，讓使用者能在「任務大廳」、「一鍵空檔速配」、「發布任務 (Demo)」、「個人資料」四個區塊間切換。
2. **無損整合 UI**：動態渲染任務卡片時，必須**完美沿用** Stitch 產出的卡片樣式與 Tailwind Class。
3. **一鍵空檔速配 (核心亮點)**：
   - 介面需有行事曆空檔選擇與一組標籤按鈕（例如：#跑腿, #文書, #體力活）。
   - 點擊送出後，前端即時根據標籤過濾出符合條件的任務。
4. **資料結構應用 (評分關鍵)**：在一鍵空檔速配邏輯中，過濾後的任務必須明確使用 **Priority Queue** 處理不衝堂任務的最高 CP 值排序，並在程式碼加上詳細中文註解。
5. **前端發布模擬 (測試沙盒)**：為確保不帶後端的發布功能不顯突兀，在「發布任務 (Demo)」頁面需有明確的「開發者測試用」提示標語，並建議預設帶入假資料（例如：[緊急] 幫買學餐）。點擊送出後，需跳出「模擬發布成功」的 UI 回饋，將新任務即時新增至 `jobs.js` 的陣列中，並自動切換回「任務大廳」展示置頂的新任務。

## Do Not Do (嚴格禁止)
- **禁止覆寫樣式**：絕對不要自己新建 `style.css` 或是寫 `<style>` 標籤去覆蓋 Stitch 的 Tailwind 設定。
- **禁止後端開發**：絕對不要寫 Node.js/Python 伺服器，或連線 Firebase/MySQL 等真實資料庫。
- **禁止無效的 UI**：畫面上不可以有任何點擊無反應的假按鈕或未完成的功能，也不要留下測試用的無意義文字。
- **禁止髒 Code**：不要把大量 JavaScript 全部塞在 HTML 裡，也不要留下大量沒用到的程式碼。

## 驗收標準 (Acceptance Criteria)

### A. 核心流程展示 (Happy Path)
> 💡 **註：此為保證可成功展示的 Demo Path，且整體展示須能在 3 分鐘內流暢完成。任何人直接打開 `index.html` 就能看到完整畫面，瀏覽器 console 不應該有會影響功能的錯誤。**

1. **頁面導覽**：點擊選單能順暢切換四個頁面，畫面不會閃爍或報錯。
2. **任務池載入**：進入「任務大廳」能成功渲染 `jobs.js` 內的 10 筆預設任務。
3. **速配觸發**：在「一鍵空檔速配」頁面勾選標籤並送出，系統透過標籤比對後，將結果成功排入行事曆空檔中。
4. **發布任務模擬 (Demo展示)**：在「發布任務 (Demo)」頁面，點擊已預填好假資料的發布按鈕，系統跳出「模擬發布成功」回饋後自動切換至「任務大廳」，並能在最上方看見剛模擬發布的新任務。

### B. 防呆與邊界處理 (Edge Cases)
5. **空值防禦**：若在速配排程或發布任務未填寫必填欄位即送出，需有前端防呆提示（如：「請至少選擇一個標籤」），不可出現 Error。
6. **多重標籤篩選正確性**：當使用者選擇多個標籤時，系統應能正確過濾出符合條件的任務，確保篩選邏輯不中斷。
7. **系統穩定性**：
   - 重新整理頁面後仍可以正常使用。
   - 操作順序稍微不同時，系統不應直接壞掉。
   - Demo 前應至少完整跑過 3 次主要流程。

### C. 技術與演算法驗證 (拿分重點)
8. **資料結構視覺化**：執行任務排程時，瀏覽器 Console (F12) 必須印出 `Priority Queue` 運作時的 Log（如：「任務依 CP 值排序完成：任務 A -> 任務 B」）。
9. **不衝堂驗證**：排出的任務時間絕對不可與行事曆上既有的「上課時間」重疊。
10. **資料結構問答準備**：程式碼中必須有清楚註解，團隊也必須能在 Demo 時流暢回答以下四個問題：
    - 使用了哪一種資料結構？ (Priority Queue)
    - 資料是怎麼存的？
    - 為什麼這裡適合用這種資料結構？
    - 這個資料結構如何影響功能運作？
